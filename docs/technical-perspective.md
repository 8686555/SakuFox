# 技术视角：AI 问数为什么不只是 NL2SQL

如果把 AI 问数理解为“自然语言输入，AI 生成 SQL，返回查询结果”，它最终会退化成一个更友好的查询入口。这个入口当然有用，但它解决的是“怎么查”，不是“该怎么理解业务”。

从技术上看，真实的分析系统远不止一个 SQL 生成器。SakuFox 的设计里，真正重要的是一条完整的数据分析链路：用户提问，系统组织上下文，Agent 选择 SQL 或 Python 工具，执行后生成结论，再把这次分析过程沉淀回工作空间，变成下一次推理可复用的经验。

这个链路里最关键的不是模型会不会写 SQL，而是它能不能拿到正确的上下文。项目里把上下文拆成了几层：数据库表结构、上传文件、知识库、挂载的 skill、会话 patch、历史迭代摘要，以及最终报告的摘要和元信息。换句话说，模型不是在真空里回答问题，而是在一个被显式建模的业务环境里推理。

从实现上看，这种思路体现在几个核心接口里：

- `/api/chat/iterate` 负责单轮分析，流式输出思考、步骤、结果和数据。
- `/api/chat/auto-analyze` 负责多轮自动分析，直到模型停止调用工具或达到轮次上限。
- `/api/chat/feedback` 负责把用户修正沉淀为业务知识或 session patch。
- `/api/skills/save` 负责把一次成功分析提炼成可复用经验，并保存 context snapshot。
- `/api/reports/iterations/{iteration_id}` 负责把自动分析报告作为可独立访问的产物提供出来。

这里真正值得强调的是反馈回路。用户指出“口径不对”“维度不对”“这个解释缺了业务背景”，系统不应该只是把这句话当成聊天记录，而应该把它转成可复用的规则、知识或补丁。这样，下一次分析同类问题时，模型不只是重新算一遍，而是带着组织已经验证过的经验继续推理。

这也是为什么报告在这个项目里不是附属物。`auto-analyze` 并不只输出结果，它还会把轮次、结论、行动建议、图表绑定和 HTML 报告一起落库。报告是这次分析的可交付物，也是后续提炼 skill 的输入材料。对技术系统来说，这一点很重要，因为它把推理过程变成了可追溯、可复用、可继承的对象。

如果只做 NL2SQL，系统的目标只是“尽快给出一个能跑的查询”。如果把整个回路做完整，系统的目标就会变成“让组织知识持续增厚”。前者是工具，后者才开始接近平台。

SakuFox 目前做的，就是尽量把这条回路落到代码里，而不是停留在概念里。
